# 电商智能客服知识库 · 详细使用说明

> 一份面向**开发者 / 运维 / 二次开发**的完整使用手册。覆盖环境准备、配置、启动、接口调用、前端使用、排错与部署。

---

## 1. 项目简介

本系统是一个 **Agentic RAG（自主检索增强生成）** 电商智能客服。核心能力：

- **知识库问答**：Milvus 向量检索 + Elasticsearch 关键词检索，RRF 重排，回答可溯源。
- **工具调用**：Agent（LangGraph 编排）自主决定是否查订单 / 查物流 / 发起退款。
- **流式对话**：基于 SSE 的逐字输出，前端 React 界面实时渲染。
- **短期记忆**：Redis 保存多轮上下文（带截断与过期）。
- **Mock 降级**：未配置 LLM Key 时仍能跑通主链路，便于无密钥环境下的联调与验证。
- **多模态预留**：`ChatDto.imageUrls` 已预留商品图上传字段（对应课程第 40 章）。

技术栈：**Nest.js 10 + LangChain.js 1.x + LangGraph 1.x + Milvus + Elasticsearch + PostgreSQL + Redis + React(Vite)**。

## 2. 环境要求

| 依赖 | 版本 | 说明 |
|------|------|------|
| Node.js | 22.x | 前端 / 后端均依赖 |
| npm | 10.x | 安装依赖（后端必须用 `--legacy-peer-deps`） |
| Docker / Docker Compose | 最新 | 启动 Milvus / ES / PG / Redis |
| OpenAI 兼容 Key | — | 真实问答需要；没有则走 Mock |

> **为什么 `--legacy-peer-deps`？** LangChain 生态已到 core 1.x，部分包（如 `better-sqlite3`）的 peer 声明与新版不兼容，普通 `npm install` 会报 `ERESOLVE`。务必加该参数。

## 4. 配置说明（`.env`）

复制模板后填写：

```bash
cp .env.example .env
```

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `PORT` | `3000` | 后端服务端口 |
| `LLM_API_KEY` | `sk-xxx` | LLM Key；**留空或设 `USE_MOCK=true` 即进入 Mock 模式**（兼容旧名 `OPENAI_API_KEY`） |
| `LLM_BASE_URL` | `https://api.openai.com/v1` | OpenAI 兼容网关（可换 DeepSeek / 通义 / 本地 Ollama 代理） |
| `LLM_MODEL` | `gpt-4o-mini` | 对话模型 |
| `LLM_EMBED_MODEL` | `text-embedding-3-small` | 向量化模型（维度需与 `MILVUS_DIM` 一致） |
| `USE_MOCK` | `false` | `true` 强制 Mock 模式 |
| `MILVUS_ADDRESS` | `localhost:19530` | Milvus 地址 |
| `MILVUS_COLLECTION` | `cs_kb` | 集合名 |
| `MILVUS_DIM` | `1536` | 向量维度（需匹配 embedding 模型） |
| `ES_NODE` | `http://localhost:9200` | ES 地址 |
| `ES_INDEX` | `cs_kb` | 索引名 |
| `PG_HOST/PORT/USER/PASSWORD/DATABASE` | `localhost/5432/postgres/postgres/cs_kb` | 业务库连接 |
| `REDIS_HOST/PORT` | `localhost/6379` | 短期记忆 |

> **Mock 判定逻辑**（`env.config.ts`）：`USE_MOCK === 'true'` **或** `LLM_API_KEY`（含兼容旧名 `OPENAI_API_KEY`）未设置 → 进入 Mock。Mock 下不调用任何 LLM / 向量库，仍返回合理的流式文本，用于验证整体结构与前端联调。

## 5. 快速开始（四种场景）

### 场景 A：纯 Mock 模式（最快，零外部依赖）

不需要 Docker、不需要 Key，仅验证代码可跑、SSE 可通：

```bash
cp .env.example .env          # 保持 LLM_API_KEY 为空 / USE_MOCK=true
npm install --legacy-peer-deps
USE_MOCK=true npm run start:dev
# 另开终端
cd frontend && npm install && npm run dev
```

打开 http://localhost:5173 即可对话（前端走 Vite 代理到 `:3000`）。

### 场景 B：完整本地（真实 LLM + 中间件）

```bash
cp .env.example .env          # 填入真实 LLM_API_KEY（可指向任意 OpenAI 兼容网关）
docker compose up -d          # 起 Milvus / ES / PG / Redis
npm install --legacy-peer-deps
npm run start:dev             # 后端 :3000
cd frontend && npm install && npm run dev   # 前端 :5173
```

### 场景 C：Docker 一体启动（含后端镜像）

```bash
cp .env.example .env
docker compose up -d          # api 服务会按 Dockerfile 构建并随中间件一起起
```

> 注意：镜像内 `npm install` 同样使用 `--legacy-peer-deps`（见 Dockerfile）。

### 场景 D：只跑前端（对接已有后端）

修改 `frontend/vite.config.ts` 的 `server.proxy['/api'].target` 指向你的后端地址，然后 `npm run dev`。

## 6. 中间件说明

`docker compose up -d` 启动 4 个中间件 + 可选 api：

| 服务 | 镜像 | 端口 | 用途 |
|------|------|------|------|
| `milvus` | milvusdb/milvus:latest (standalone) | 19530 | 向量检索 |
| `elasticsearch` | elasticsearch:8.14.0 | 9200 | 关键词 BM25 |
| `postgres` | postgres:16 | 5432 | 业务库（订单/物流等） |
| `redis` | redis:7 | 6379 | 短期记忆 |

数据均挂载到命名卷（`milvus_data` / `es_data` / `pg_data` / `redis_data`），重启不丢。

> **首次启动 Milvus 稍慢**（需拉镜像 + 初始化），可用 `docker compose logs -f milvus` 观察就绪。

## 8. 知识库入库（Ingest）

把文档写入向量库 + 关键词库（自动切分，chunkSize≈500）。

**请求**

```bash
curl -X POST http://localhost:3000/api/ingest \
  -H 'Content-Type: application/json' \
  -d '{
    "title": "七天无理由退换货政策",
    "text": "商品签收后 7 天内可申请无理由退货，需保持吊牌完好、不影响二次销售……",
    "sourceType": "MD"
  }'
```

**参数**（IngestDto）**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `title` | string | 是 | 文档标题 |
| `text` | string | 是 | 正文 |
| `sourceType` | string | 否 | 来源类型（PDF/WORD/HTML/MD），仅标注用 |

> 入库成功后会同时写入 Milvus（向量）与 Elasticsearch（关键词），供后续混合检索使用。多份文档可多次调用。

## 9. 对话接口

### 9.1 流式对话（SSE）— 前端主用

```bash
curl -N -X POST http://localhost:3000/api/chat/stream \
  -H 'Content-Type: application/json' \
  -d '{"sessionId":"s1","message":"我买的鞋子什么时候到？订单号 SO123"}'
```

- 方法：`POST`，body 为 `ChatDto`
- 返回：`text/event-stream`，每条事件 `data: <文本片段>` 由空行分隔
- 前端用 `fetch + ReadableStream` 手动解析（浏览器原生 `EventSource` 仅支持 GET）

**ChatDto 参数**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `sessionId` | string | 是 | 会话 ID，用于关联 Redis 多轮记忆 |
| `message` | string | 是 | 用户问题 |
| `imageUrls` | string[] | 否 | 多模态图片 URL（预留） |

### 9.2 非流式对话

```bash
curl -X POST http://localhost:3000/api/chat \
  -H 'Content-Type: application/json' \
  -d '{"sessionId":"s1","message":"退换货政策是什么？"}'
# {"answer":"..."}
```

### 9.3 响应中的引用与工具数据

- 真实模式下，Agent 可能调用 `order_status` 等工具，其返回的 JSON 会作为上下文进入回答；前端会把 ` ```json ` 代码块渲染为「工具返回 / 数据」卡片。
- 引用溯源标记（如 `【政策文档】` 或 `[引用:...]`）会被前端识别为可溯源 chip。

## 11. Mock 模式 vs 真实模式

| 维度 | Mock 模式 | 真实模式 |
|------|-----------|----------|
| 触发 | `USE_MOCK=true` 或无 Key | 配置 `LLM_API_KEY`（兼容旧名 `OPENAI_API_KEY`） |
| LLM | 不调用，返回固化示例文本 | 调用配置的对话/embedding 模型 |
| Milvus / ES | 不连接 | 连接并做混合检索 |
| 工具 | 不调用（示例文本提及订单状态） | Agent 自主调用 order/logistics/refund |
| 用途 | 结构验证、前端联调、无网环境 | 真实问答效果 |

## 12. 常见问题（FAQ）

**Q1：`npm install` 报 ERESOLVE / peer 冲突？**
→ 必须加 `--legacy-peer-deps`。

**Q2：启动后 SSE 报错 / Redis ECONNREFUSED？**
→ 记忆层对 Redis 失败做了**容错降级**（try-catch + 静默），无 Redis 也能跑通主链路；生产请确认 `docker compose up -d` 已起 redis。

**Q3：Milvus 连接不上？**
→ 确认 `docker compose up -d` 已起 milvus，且 `MILVUS_ADDRESS` 与容器端口一致（默认 `localhost:19530`）。首次启动 Milvus 需等待就绪。

**Q4：维度不匹配（dimension mismatch）？**
→ `MILVUS_DIM` 必须与 `LLM_EMBED_MODEL` 输出维度一致（如 `text-embedding-3-small` = 1536）。换 embedding 模型时同步修改。

**Q5：工具查订单返回空？**
→ 订单/物流/退款工具查询的是 **PostgreSQL 业务表**（`orders` / `logistics` / `refunds`）。当前仓库未内置建表 SQL 与种子数据，需自行在 `cs_kb` 库创建这些表并插入示例数据后，工具才会返回真实结果。Mock 模式下不影响服务可用性。

**Q6：前端连不上后端？**
→ 确认后端在 `:3000` 已起；开发态依赖 Vite 代理，确认 `vite.config.ts` 的 `target` 正确。

## 13. 部署建议

- **后端**：`docker compose up -d` 一体部署，或单独构建 `Dockerfile` 推到云服务器 / 容器服务。
- **前端**：`npm run build` 后由 Nginx / Vercel / 对象存储托管静态资源，Nginx 反代 `/api` 到后端。
- **安全性**：
  - `.env` 含 API Key，**切勿提交**（已在 `.gitignore`）。
  - 公网部署务必启用鉴权（如 API Key / JWT）与速率限制，避免被刷。
  - 若用 OpenAI 兼容网关，注意流量与费用监控。

## 14. 索引可维护性与模型切换

向量索引维度在 Milvus 中**建好即锁死**，所有主流向量库（Milvus / Pinecone / pgvector）均如此。本项目据此做了两项加固：

### 14.1 启动期维度自检（fail-fast）

非 Mock 模式下，服务启动会比对「embedding 模型实际输出维度」与「Milvus 集合维度」：

- 一致 → 打印 `[dimension] OK`
- 集合不存在 → 跳过（首次入库会按正确维度建集合）
- **不一致 → 明确报错并阻断启动**，给出两种重建指引：
  1. 改 `MILVUS_COLLECTION` 为带维度后缀的新名（如 `cs_kb_3072`），重启自动建新集合；
  2. 在 Attu 删除旧集合后重新入库。

> Milvus / PG 临时不可达时自检会自动跳过（仅告警），不阻断启动。

### 14.2 原始文档持久化 + 一键重灌

入库时会把 `{title, text, source_type, embedding_model}` 写入 **PostgreSQL `kb_documents` 表**（幂等建表）。向量库与 ES 被视为**可重建的派生索引**，原始文本才是唯一真相。

切换 embedding 模型后的标准流程：

```bash
# 1) 改 .env：LLM_EMBED_MODEL=text-embedding-3-large
# 2) 避免维度冲突：改 MILVUS_COLLECTION=cs_kb_3072（或 Attu 删旧集合）
# 3) 重启后端（维度自检通过，新建空集合）
# 4) 一键重灌（读取 kb_documents，用当前模型重新向量化）
curl -X POST http://localhost:3000/api/ingest/reindex
# => {"reindexedDocs": N}
```

`reindex` 会向目标集合**追加**文档；若需完全干净，请先确保集合为空（新集合名 / 已删旧集合）。
